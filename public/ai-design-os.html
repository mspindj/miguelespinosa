The AI Design
Operating System

A five-component framework for running product design work when AI is in the room and you need it to do what you expect.

Miguel Espinosa  ·  June 2026  ·  15 min read

There's a version of using AI that feels like typing into a void. You get something back, you iterate, it's fine. It doesn't compound. Every session starts the same as the last.

I ran into this across my latest projects. Each one different context, same failure mode: no environment around the model, so nothing persisted, nothing got smarter, nothing was traceable. When something went wrong, you had no way to know where.

What fixed it wasn't a better model. It was five files and three agent roles. That's this guide.

I stopped switching models about six months in. Nothing I tried made as much difference as tightening the context files.

The Framework

Five components. One afternoon to set up.

Think of it as the environment the model operates inside. What it reads before it acts. What it's allowed to touch. What happens after it finishes. The prompt is part of that environment, but a small part.

01

Context Files

What the model reads before it acts. DESIGN.md captures your visual system. voice.md defines how the AI sounds and handles uncertainty. domain-patterns.md encodes the industry knowledge no model can infer. Without these, the model invents its own version of your product.

DESIGN.md voice.md domain-patterns.md
02

Agent Roles

The agent that writes the spec shouldn't write the code. Separation of concerns applies to agents too. One agent to specify, one to implement, one to review. Each role gets its own context, its own constraints, its own definition of done.

specifier implementer reviewer
03

Memory

The model doesn't remember you. Your files do. corrections.md captures every time the model did something wrong and what the right answer was. dont-do.md captures approaches you've explicitly rejected. Both travel with the project forever.

corrections.md dont-do.md
04

Validation

Checkpoints that run after the model acts. Did it touch files outside task scope? Stop. Does the output match what was specified? Verify. Validation is how you It sounds paranoid until the first time the model quietly refactors something you didn't ask it to touch.

scope checks output verification quality gates
05

Human Gates

Deliberate pauses where you decide what happens next. The model can keep going without you — sometimes that's fine. For decisions that are hard to reverse, build a pause before them.

review checkpoints approval workflows
The New Artifacts

What you actually produce now

The deliverables changed. There are four files you'll need that didn't exist in your workflow two years ago.

DESIGN.md

Your design system, written for the model to read. Semantic color tokens, spacing scale, type scale, component inventory. Without this file, the model generates buttons that are technically correct and visually wrong.

voice.md

How the AI sounds in your product. How confident it is at different confidence levels. What it says when it doesn't know. How it handles errors. Every AI-generated string in the product has a voice — this file decides what that voice is.

domain-patterns.md

Industry knowledge the model can't infer. In a golf app: what's the difference between a handicap and a playing handicap? In banking: what does "cash conversion" actually mean to a retail customer? Domain depth lives here.

corrections.md

An append-only log of every agent mistake and its correction. Date, context, what went wrong, what the right behavior is. No measurable ROI on day one. By month three, it's the most valuable artifact in your project.

Template

A minimal DESIGN.md to start from

Copy this into your project root and fill in your actual values. The only rule: use semantic token names. Not hex values, not Tailwind class names. The model needs to know what a color means, not what it looks like. 

# DESIGN.md

## Grid
Desktop: 12 columns, 32px margin, 24px gutter
Mobile: 4 columns, 16px margin, 16px gutter

## Spacing
Base unit: 8px
spacing-1: 8px | spacing-2: 16px | spacing-3: 24px
spacing-4: 32px | spacing-6: 48px | spacing-8: 64px

## Typography
Display: 48px / 700 — hero headings only
H1: 32px / 600 | H2: 24px / 600 | H3: 20px / 500
Body: 16px / 400 | Small: 14px / 400 | Caption: 12px / 400

## Color tokens
primary / primary-foreground   — main actions, CTAs
secondary / secondary-foreground — supporting actions
destructive / destructive-foreground — delete, irreversible
muted / muted-foreground       — disabled, secondary text
card / card-foreground         — card surfaces
background / foreground        — page background, body text

## Radius
--radius: 8px
sm: calc(var(--radius) - 2px)
md: var(--radius)
lg: calc(var(--radius) + 4px)

## Restrictions
Never use raw hex values — always semantic tokens
Never override spacing outside the scale
Card elevation: box-shadow only, no border on light theme
Getting Started

Your first week

Don't try to build the system all at once. Build it in the order things become painful. This is the sequence that works.

Day 1
Write DESIGN.md

Two hours. Semantic tokens, spacing, type scale, component list. It will pay back in weeks.

Day 2
Write your first corrections.md entry

Find something the model got wrong today. Document it. The file exists now. It will compound.

Day 3
Define three agents

Specifier. Implementer. Reviewer. They don't have to be sophisticated. They have to be separate.

Day 4
Add your first human gate

One checkpoint where you approve before the agent continues. One.

Day 5
Write domain-patterns.md

What does your product domain know that the model doesn't? Write that down.

Day 7
You have a system

Rough at the edges. It'll get sharper as you correct it.

Who This Is For

Check before you invest the week

This is for you if

You're already using AI tools and feel like you're losing control
You spend more time correcting the AI than using it
You're a designer or design leader running real product work
You want a system you can set up this week, not a theory for next quarter

This is not for you if

You're looking for a technical architecture guide
You haven't started using AI in your workflow yet
You want a framework someone else built that you can just install
